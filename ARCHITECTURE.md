# Архитектурная карта проекта: MangaEditor

## 1. Стек технологий и паттерны
* **Язык/Фреймворк:** Python 3.10+ / PySide6 (Qt for Python), PyTorch, OpenCV, ONNX Runtime
* **База данных:** N/A (файловое хранилище графических проектов и временных оверлеев)
* **Архитектурный подход:** Модульный GUI на PySide6 с декларативным пайплайном обработки изображений (U-Net сегментация -> гибридный LaMa MPE inpainting -> FFT-based донорная заливка текстур скринтонов).

## 2. Карта модулей и функций (Локация кода)

* `[main.py]` — Точка входа графического интерфейса пользователя, управление слоями, холстом и Undo/Redo стеком.
  * `cv_imread_unicode()` [строки 40-46] — Загрузка изображений из путей с кириллицей через cv2.imdecode.
  * `cv_imwrite_unicode()` [строки 48-59] — Сохранение изображений с кириллическими путями через cv2.imencode.
  * `MainWindow.__init__()` [строки 1080-1200] — Инициализация главного окна GUI, меню, виджетов управления и связывание сигналов.
  * `MainWindow.clean_selected_area()` [строки 1400-1450] — Вызов гибридной очистки выделенного прямоугольника или текстовых бабблов.

* `[lama_mpe_pytorch.py]` — Движок нейросетевого инпейнгинга LaMa MPE с встроенным детектором контуров и сегментатором U-Net.
  * `load_lama_mpe()` [строки 580-598] — Загрузка весов чекпоинта LaMa MPE в модель PyTorch.
  * `LamaMPEPyTorchInpainter.__init__()` [строки 601-618] — Инициализация класса инпейнтера и загрузка модели сегментатора `segmenter.onnx`.
  * `LamaMPEPyTorchInpainter.inpaint()` [строки 619-831] — Главный гибридный пайплайн очистки: детекция букв U-Net + ConnectedComponents, ультра-точечная дилатация маски, LaMa inpainting, локальная донорная заливка скринтонов в окне context expansion (+128px) и прямая подстановка пикселей без feather-размытия.

* `[fast_exemplar_inpaint.py]` — Алгоритм донорной заливки текстур скринтонов без нейросетей на основе FFT-анализа.
  * `detect_period_fft()` [строки 5-68] — Вычисление 2D-периода паттерна скринтона (dy, dx) через спектральный автокорреляционный анализ FFT.
  * `periodic_fill()` [строки 71-124] — Быстрое PatchMatch-заполнение маски периодическими сдвигами по найденному вектору периода с фильтрацией донорных контуров.
  * `nearest_neighbor_fill()` [строки 126-144] — Запасное мгновенное заполнение невосполненных пикселей маски по матрице расстояний (distanceTransformWithLabels).
  * `feather_blend()` [строки 147-154] — Сглаживание границ между результатом заливки и оригиналом по краям маски.
  * `fast_exemplar_inpaint()` [строки 157-198] — Основная функция донорной заливки, объединяющая FFT-детектор, периодическое заполнение, защиту контуров и краевой блендинг.

## 3. Реестр архитектурных решений (ADR)
* **2026-07-21 — Решение №1: Гибридный двухэтапный пайплайн очистки (LaMa + Fast Exemplar FFT)**
  * **Контекст:** Обычный LaMa PyTorch размывает периодические точки скринтонов манги в мыльные серые пятна, а чистый Fast Exemplar порвал бы сложный лайн-арт одежды и волос.
  * **Почему принято:** LaMa идеально восстанавливает базовую структуру (белый цвет одежды, границы объектов, сложные градиенты), а Fast Exemplar FFT поверх нее заменяет чернила букв точным периодическим паттерном точек.
  * **Альтернативы:** Использование GAN-моделей генерации текстур (слишком медленно, требует GPU) или ручного штампа (требует постоянной работы пользователя).

* **2026-07-21 — Решение №2: Расширение контекста донора (Context Expansion +128px)**
  * **Контекст:** При маленьком прямоугольном выделении донору не хватало чистых пикселей скринтона вокруг маски для корректного FFT-анализа.
  * **Почему принято:** Границы вырезки для FFT расширяются наружу на 128px во все стороны, обеспечивая донорную область чистым скринтоном без изменения внешней части картинки.
  * **Альтернативы:** Проведение FFT по всей странице (слишком медленно и берет чужие скринтоны с других панелей).

* **2026-07-21 — Решение №3: Ультра-точечная дилатация маски донора (1px) и изоляция плоских фонов**
  * **Контекст:** При дилатации маски на 5px маска сливалась в один большой блок во весь кроп, из-за чего донор стирал оригинальный скринтон между буквами и создавал полосы.
  * **Почему принято:** Дилатация для донора скринтона снижена до 1px (только сама тушь букв + микро-граница). 95% оригинального скринтона между буквами остается нетронутым. Донор вызывается только при `bg_mean < 205` (серый скринтон), а на белых рубашках работает только чистый LaMa.
  * **Альтернативы:** Сплошная заливка всего пользовательского выделения (разрушает оригинальный рисунок художников).

* **2026-07-21 — Решение №4: Морфологическая эрозия карты заблокированных контуров (donor_edges)**
  * **Контекст:** Тонкие линии эффектов (speedlines) на скринтоне блокировались как `donor_edges`, оставляя донора без единого чистейшего пикселя для копирования.
  * **Почему принято:** Карту `donor_edges` пропускаем через эрозию (`cv2.erode`). Тонкие линии эффектов (1-2px) вымываются и разрешают брать скринтон вокруг них, а толстые линии рисунка (воротник, рама кадра) сохраняются и блокируют донор от попадания чернил рисунка на текстуру.
  * **Альтернативы:** Отключение фильтрации контуров (копирует черные волосы и воротник на скринтон).

* **2026-07-21 — Решение №5: Прямая точечная замена пикселей туши без feather-размытия**
  * **Контекст:** Функция `feather_blend` смешивала четкие бинарные точки скринтона с мыльным серым выходом LaMa по краям маски, создавая видимые серые полосы и ореолы вокруг выделенных областей.
  * **Почему принято:** Результат донора вклеивается строго по пикселям маски (`screentone_bool`) без полупрозрачного размытия краев, что сохраняет 100% бинарную четкость точек скринтона.
  * **Альтернативы:** Гауссово размытие маски (создает мыльный серый шлейф вокруг растровых точек).
